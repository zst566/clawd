## 二、高危问题

### 1. 开发环境测试账号硬编码泄露 【高危】

**问题位置：**
- `apps/h5-client/src/pages/login/index.vue` (第 41-47 行)
- `apps/h5-channel/src/pages/login/index.vue` (第 41-47 行)
- `apps/h5-teacher/src/pages/login/index.vue` (第 41-47 行)

**问题描述：**
三个登录页面均包含硬编码的开发环境测试账号，直接显示在前端页面中：

```vue
<div class="dev-tip">
  <p>开发环境测试账号</p>
  <p>用户名: test_user / channel_001 / teacher_001</p>
  <p>密码: 123456</p>
</div>
```

**风险分析：**
- 测试账号暴露在生产环境可能被恶意利用
- 攻击者可利用这些账号进行未授权访问
- 即使在开发环境展示，也容易被截图或爬取

**修复建议：**
```vue
<!-- 生产环境完全移除测试账号显示 -->
<div v-if="isDevEnvironment" class="dev-tip">
  <p>开发环境测试账号</p>
  <p>用户名: test_user</p>
  <p>密码: 123456</p>
</div>
```

或使用环境变量控制：
```typescript
const showDevTip = import.meta.env.DEV && import.meta.env.VITE_SHOW_DEV_TIP === 'true'
```

### 2. Token 存储在 localStorage 【高危】

**问题位置：**
- `apps/h5-client/src/stores/user.ts` (第 9, 17-19 行)
- `apps/h5-channel/src/stores/user.ts` (第 9, 17-19 行)
- `apps/h5-teacher/src/stores/user.ts` (第 9, 17-19 行)

**问题描述：**
所有三个端都将 JWT Token 存储在 localStorage 中：

```typescript
const token = ref<string | null>(localStorage.getItem('token'))

function setToken(newToken: string) {
  token.value = newToken
  localStorage.setItem('token', newToken)
}
```

**风险分析：**
- localStorage 无法防止 XSS 攻击，恶意脚本可读取 Token
- 攻击者可通过 XSS 漏洞窃取 Token 并冒充用户
- localStorage 数据可被其他同源页面读取

**修复建议：**
使用 HttpOnly Cookie 存储 Token，或使用 memoryStorage + 刷新令牌机制：

```typescript
// 方案1：使用 Cookie（推荐）
function setToken(newToken: string) {
  // 设置 HttpOnly Cookie（需后端配合）
  document.cookie = `token=${newToken}; HttpOnly; Secure; SameSite=Strict; Path=/`
}

// 方案2：内存存储 + 定期刷新
const token = ref<string | null>(null)  // 不再从 localStorage 读取

function setToken(newToken: string) {
  token.value = newToken
  // 可选：加密存储到 localStorage 作为备份
  localStorage.setItem('encrypted_token', encrypt(newToken))
}
```

### 3. 密码明文传输 【高危】

**问题位置：**
- `apps/h5-client/src/api/request.ts`
- `apps/h5-channel/src/api/request.ts`
- `apps/h5-teacher/src/api/request.ts`

**问题描述：**
登录请求使用 JSON 明文传输密码，未进行任何加密：

```typescript
const res = await devLogin({
  username: username.value,
  password: password.value,  // 明文传输
  clientType: 'user',
})
```

请求头配置：
```typescript
headers: {
  'Content-Type': 'application/json',
}
```

**风险分析：**
- HTTP 请求可被网络层窃听（中间人攻击）
- 代理服务器、日志系统可能记录明文密码
- 在公共 WiFi 环境下尤其危险

**修复建议：**
1. **强制 HTTPS**：确保生产环境全部使用 HTTPS
2. **密码加密**：使用 RSA 公钥加密密码后再传输
3. **使用 Authorization 头**：避免将凭据放在请求体中

```typescript
// 前端使用公钥加密密码
import { encryptPassword } from '@/utils/crypto'

const encryptedPassword = await encryptPassword(password.value, publicKey)

const res = await devLogin({
  username: username.value,
  password: encryptedPassword,
  clientType: 'user',
})
```

## 三、中危问题

### 4. 缺少 CSRF 防护 【中危】

**问题位置：**
- 所有 `request.ts` 文件

**问题描述：**
未发现 CSRF Token 的生成、存储和使用机制。请求拦截器仅添加了 Authorization 头：

```typescript
instance.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('token')
    if (token) {
      config.headers.Authorization = `Bearer ${token}`
    }
    return config
  },
  // 缺少 CSRF token
)
```

**风险分析：**
- 攻击者可诱导已登录用户访问恶意页面
- 利用用户的登录态发起跨站请求
- 特别是涉及敏感操作时风险更高

**修复建议：**
```typescript
// 请求拦截器添加 CSRF Token
instance.interceptors.request.use(
  (config) => {
    const csrfToken = localStorage.getItem('csrf_token')
    if (csrfToken && ['post', 'put', 'delete'].includes(config.method?.toLowerCase() || '')) {
      config.headers['X-CSRF-Token'] = csrfToken
    }
    return config
  },
)
```

### 5. 开发环境登录接口暴露 【中危】

**问题位置：**
- `apps/h5-client/src/api/auth.ts`
- `apps/h5-channel/src/api/auth.ts`
- `apps/h5-teacher/src/api/auth.ts`

**问题描述：**
三个端都使用 `devLogin` 接口进行登录，这是一个开发环境专用的登录方式：

```typescript
export function devLogin(params: DevLoginParams): Promise<ApiResponse<LoginResponse>> {
  return post('/auth/dev-login', params)
}
```

**风险分析：**
- 生产环境可能仍保留此接口
- 攻击者可尝试使用此接口绕过正常登录流程
- 开发接口通常安全加固较弱

**修复建议：**
1. 生产环境禁用 dev-login 接口
2. 通过环境变量控制接口可用性
3. 添加 IP 白名单限制

```typescript
export function devLogin(params: DevLoginParams): Promise<ApiResponse<LoginResponse>> {
  if (!import.meta.env.DEV) {
    throw new Error('开发环境登录仅限本地使用')
  }
  return post('/auth/dev-login', params)
}
```

### 6. 缺少登录尝试限制 【中危】

**问题位置：**
- 登录页面 `handleLogin` 函数

**问题描述：**
未发现验证码、滑动验证或登录失败次数限制机制：

```typescript
async function handleLogin() {
  if (loading.value) return
  loading.value = true
  // 直接发送登录请求，无防暴力破解机制
  const res = await devLogin({...})
}
```

**风险分析：**
- 攻击者可无限尝试密码爆破
- 容易被暴力破解弱密码账户
- 没有账户锁定机制

**修复建议：**
1. 引入图形验证码或滑动验证
2. 限制单位时间内登录尝试次数
3. 多次失败后锁定账户

```typescript
const MAX_LOGIN_ATTEMPTS = 5
const LOCKOUT_duration = 15 * 60 * 1000  // 15分钟

async function handleLogin() {
  if (loading.value) return
  
  // 检查是否被锁定
  const lockUntil = localStorage.getItem('login_lock_until')
  if (lockUntil && Date.now() < parseInt(lockUntil)) {
    showToast({ message: '登录尝试过多，请稍后再试', type: 'fail' })
    return
  }
  
  loading.value = true
  // ... 登录逻辑
}
```

### 7. Token 无过期时间管理 【中危】

**问题位置：**
- 各端 `stores/user.ts`

**问题描述：**
Token 存储后无过期时间检查和自动刷新机制：

```typescript
function setToken(newToken: string) {
  token.value = newToken
  localStorage.setItem('token', newToken)
  // 缺少过期时间存储
}
```

**风险分析：**
- 长时间有效的 Token 被窃取后危害更大
- 无法实现"最近活动后延长过期时间"逻辑
- 用户无法主动使已泄露的 Token 失效

**修复建议：**
```typescript
interface TokenData {
  token: string
  expiresAt: number  // 过期时间戳
  refreshAt: number  // 提前刷新时间
}

function setToken(tokenData: string, expiresIn: number = 7200) {  // 默认2小时
  const now = Date.now()
  const data: TokenData = {
    token: tokenData,
    expiresAt: now + expiresIn * 1000,
    refreshAt: now + (expiresIn - 600) * 1000,  // 提前10分钟刷新
  }
  localStorage.setItem('token_data', JSON.stringify(data))
}

function isTokenExpired(): boolean {
  const data = localStorage.getItem('token_data')
  if (!data) return true
  return Date.now() > JSON.parse(data).expiresAt
}
```

## 四、低危问题

### 8. 错误信息不够具体 【低危】

**问题位置：**
- 登录页面 `handleLogin` 函数

**问题描述：**
登录失败时仅显示通用错误信息：

```typescript
showToast({
  message: error.message || '登录失败，请检查网络',
  type: 'fail',
})
```

**风险分析：**
- 用户无法区分是账号错误、密码错误还是系统问题
- 过于通用的错误信息可能影响用户体验

**修复建议：**
根据不同错误码返回具体信息（但避免提示"用户名不存在"这种可能泄露用户存在性的信息）：

```typescript
if (error.response?.status === 401) {
  showToast({ message: '用户名或密码错误', type: 'fail' })
} else if (error.response?.status === 423) {
  showToast({ message: '账号已被锁定，请稍后再试', type: 'fail' })
} else {
  showToast({ message: '登录失败，请检查网络', type: 'fail' })
}
```

### 9. 登录后重定向缺少验证 【低危】

**问题位置：**
- 各端登录页面

**问题描述：**
登录成功后直接跳转到固定路由，未验证目标地址：

```typescript
router.replace({ name: 'home-new' })  // h5-client
router.replace({ name: 'home' })       // h5-channel / h5-teacher
```

**风险分析：**
- 虽然当前未发现 open redirect 问题
- 但如果后续添加 `redirect` 参数，可能存在风险

**修复建议：**
```typescript
function getRedirectUrl(): string {
  const urlParams = new URLSearchParams(window.location.search)
  const redirect = urlParams.get('redirect')
  
  // 验证 redirect 是相对路径且安全
  if (redirect && redirect.startsWith('/') && !redirect.startsWith('//')) {
    return redirect
  }
  return '/home'
}

router.replace(getRedirectUrl())
```

## 五、总结与建议

### 优先修复项（高危）

1. **移除生产环境测试账号显示**
2. **将 Token 存储改为 HttpOnly Cookie 或内存存储**
3. **确保生产环境强制使用 HTTPS**

### 次要修复项（中危）

4. **添加 CSRF 防护机制**
5. **生产环境禁用 dev-login 接口**
6. **添加登录尝试次数限制**
7. **实现 Token 过期时间管理**

### 优化项（低危）

8. **改进错误信息提示**
9. **添加重定向地址验证**
