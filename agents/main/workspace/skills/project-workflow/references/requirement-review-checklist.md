# 需求评审检查清单

> 用于系统性地评审需求文档，确保没有遗漏关键信息

---

## 一、业务理解

### 1.1 业务背景
- [ ] 客户公司背景和行业特点是否清楚
- [ ] 项目的业务目标是什么（解决什么问题）
- [ ] 项目的成功标准是什么
- [ ] 项目的失败风险有哪些

### 1.2 用户角色
- [ ] 是否识别了所有用户角色
- [ ] 每个角色的职责和权限是否明确
- [ ] 角色之间的协作流程是否清楚

### 1.3 业务流程
- [ ] 核心业务流程是否完整（从起点到终点）
- [ ] 是否有异常流程（错误处理、边界情况）
- [ ] 是否有替代流程（不同场景下的不同处理）
- [ ] 流程图是否清晰易懂

---

## 二、功能需求

### 2.1 功能完整性
- [ ] 是否列出了所有功能模块
- [ ] 每个功能模块的子功能是否完整
- [ ] 是否有遗漏的隐性需求

### 2.2 功能描述
- [ ] 功能描述是否清晰（无歧义）
- [ ] 功能的输入和输出是否明确
- [ ] 功能的业务规则是否清楚
- [ ] 功能的优先级是否标注（P0/P1/P2）

### 2.3 数据需求
- [ ] 核心数据实体是否识别
- [ ] 数据字段和数据类型是否明确
- [ ] 数据量预估是否合理
- [ ] 数据的生命周期是否清楚（创建、更新、删除）

---

## 三、非功能性需求

### 3.1 性能需求
- [ ] 并发用户数预估
- [ ] 响应时间要求（页面加载、接口响应）
- [ ] 吞吐量要求（QPS/TPS）

### 3.2 安全需求
- [ ] 用户认证方式（登录、权限）
- [ ] 敏感数据保护要求
- [ ] 合规要求（数据隐私、行业规范）

### 3.3 兼容性需求
- [ ] 浏览器兼容性要求
- [ ] 移动端适配要求
- [ ] 第三方系统集成要求

### 3.4 可用性需求
- [ ] 系统可用性指标（SLA）
- [ ] 备份和恢复要求
- [ ] 监控和告警要求

---

## 四、项目边界

### 4.1 范围界定
- [ ] 项目包含的功能范围是否清晰
- [ ] 项目不包含的功能是否明确（哪些不做）
- [ ] 本次版本和未来版本的划分是否清楚

### 4.2 外部依赖
- [ ] 是否有第三方系统需要对接
- [ ] 对接接口是否已明确
- [ ] 依赖方的配合程度如何

---

## 五、待确认问题

### 5.1 问题清单
- [ ] 是否列出了所有待确认问题
- [ ] 每个问题的影响范围是否清楚
- [ ] 是否有问题的建议方案

### 5.2 风险评估
- [ ] 需求中是否有高风险点
- [ ] 是否有技术可行性风险
- [ ] 是否有业务规则不明确的风险

---

## 六、评审输出

### 评审结论
- [ ] **通过**：需求清晰完整，可以进入设计阶段
- [ ] **有条件通过**：存在小问题，修正后可进入设计阶段
- [ ] **不通过**：存在重大问题，需要重新梳理需求

- 评审日期：________
- 参与人员：________
- 发现的问题：________
- 需要修正的内容：________
- 下次评审时间（如需要）：________

---

## 评审建议

1. **提前发送文档**：评审前1-2天将需求文档发送给参与者
2. **控制会议时间**：建议1-2小时，超过则分多次评审
3. **记录会议纪要用**：专人记录，会后24小时内发出
4. **聚焦关键问题**：不在细节上过多纠缠，细节可在设计阶段细化
5. **获取书面确认**：评审通过后，获取甲方书面确认
