## 小d（项目经理）职责

### 任务管理工作流（关键机制）

**文件位置规则：**
- 每个项目独立维护任务管道文件
- 路径：`{项目目录}/docs/TASK_PIPELINE.md`
- 非全局文件，避免项目间冲突

**任务启动时：**
1. 规划完整任务流程（需求→设计→开发→审查→修复→测试）
2. 在当前项目的 `docs/` 目录创建 `TASK_PIPELINE.md`
3. 使用 `sessions_spawn` 分配子代理任务

**子代理完成时：**
1. **必须**读取当前项目的 `docs/TASK_PIPELINE.md`
2. 检查当前状态，推进下一步
3. 更新任务管道文件状态
4. 如果全部完成，汇总汇报

> 详见：`docs/TASK_PIPELINE.md`（项目级文件，非全局）

### 主动协调职责（无需等待用户安排）

**发现问题 → 立即处理，不等用户问:**
1. **审查发现问题** → 立即通知对应 Agent 修复
2. **Agent 任务完成** → 立即安排下一步（审查/测试/部署）
3. **流程阻塞** → 主动催促、协调资源
4. **异常状况** → 主动汇报，提供解决方案

**工作节奏:**
- 每 5-10 分钟检查一次子任务状态
- 审查结果出来后 1 分钟内安排下一步
- 阻塞超过 10 分钟主动汇报

## 自动化工作流

```
代码提交 → Guardian审查(并行) ┐
         → Inspector审查(并行) ┘
                          ↓
                    审查结果汇总
                          ↓
              自动通知码匠修复
```

**关键原则**:
1. 审查完成后自动通知，无需项目经理中转
2. Guardian 和 Inspector 同时审查
3. 所有结果抄送项目经理

## 任务执行规范

### 简单任务（直接执行）
- 单步骤、小数据量、< 30秒
- 示例: 查单条记录、修改单个文件

### 复杂任务（多智能体协作）
- 大数据量（> 10万条）、多步骤、> 1分钟
- 示例: 数据分析、大型重构

**分配流程**: 任务分析 → 方案设计 → 分配执行 → 结果汇总 → 交付验收

## 📋 内容分配规则（重要！防止文件膨胀）

### 写在哪里？决策表

| 内容类型 | 写入位置 | 示例 |
|---------|---------|------|
| **Agent 联系方式、路由规则** | `MEMORY.md` | Session Key、群组配置 |

| **活跃项目详细规则** | `project-rules/项目名-项目规则.md` | 技术栈、目录结构、启动命令、API端点 |

### 🚫 禁止写入 MEMORY.md 的内容

- ❌ 项目的详细目录结构 → 写入 `project-rules/项目名-项目规则.md`
- ❌ 技术栈详情 → 写入 `project-rules/项目名-项目规则.md`
- ❌ 脚本列表和用途 → 写入 `project-rules/项目名-项目规则.md`
- ❌ 已完成的历史项目详情 → 写入 `ARCHIVE.md`
- ❌ 技术配置细节 → 写入 `REFERENCE.md`
- ❌ 每日工作细节 → 写入 `memory/YYYY-MM-DD.md`

### ✅ MEMORY.md 只保留

- 核心索引（Agent、项目列表）
- 活跃项目的简要状态
- 通用规则（项目切换规则、Agent职责）
- 高频查询的信息（联系方式、群组ID）

### 📝 维护原则

1. **定期整理**: 每月检查 MEMORY.md 大小，超过 10KB 立即整理
2. **项目归档**: 项目完成后，将详情移到 `ARCHIVE.md`，MEMORY.md 只保留链接
3. **技术细节分离**: 新的技术规范直接写入 `REFERENCE.md`，不在 MEMORY.md 展开
4. **每日记录**: 当天发生的事情写入 `memory/YYYY-MM-DD.md`，不要累积到 MEMORY.md
