# 项目启动工作流

> 标准化软件项目从启动到交付的完整工作流程  
> 版本：v1.0  
> 适用：Web应用、管理系统、小程序等软件项目

---

## 快速导航

| 场景 | 操作 |
|------|------|
| 新项目启动 | 从**阶段1**开始执行 |
| 需求已明确，开始设计 | 跳至**阶段2** |
| 设计已完成，开始开发 | 跳至**阶段3** |
| 开发完成，准备交付 | 跳至**阶段4** |

---

## 工作流总览

```
阶段1: 需求阶段 ─────────────────────────────────────────
├── 1.1 需求收集           【产出】需求调研笔记
├── 1.2 需求分析           【产出】需求分析文档
├── 1.3 需求评审           【产出】需求评审报告 ← ⚠️ 关键检查点
└── 1.4 需求确认           【产出】需求确认书（客户签字）

阶段2: 设计阶段 ─────────────────────────────────────────
├── 2.1 原型/UI设计        【产出】Figma/Axure原型
├── 2.2 技术架构设计       【产出】技术架构文档
├── 2.3 数据库设计         【产出】数据库设计文档
├── 2.4 API接口设计        【产出】API接口文档
└── 2.5 设计评审           【产出】设计评审报告 ← ⚠️ 关键检查点

阶段3: 开发阶段 ─────────────────────────────────────────
├── 3.1 开发环境搭建
├── 3.2 编码实现           【产出】源代码 + 单元测试
├── 3.3 代码审查
├── 3.4 联调测试
└── 3.5 测试报告           【产出】测试报告 ← ⚠️ 关键检查点

阶段4: 验收阶段 ─────────────────────────────────────────
├── 4.1 内部验收
├── 4.2 客户验收           【产出】验收报告（客户签字）← ⚠️ 关键检查点
└── 4.3 上线部署           【产出】部署文档 + 运维手册
```

## 阶段1：需求阶段

### 1.1 需求收集

**目标**：完整收集客户的业务需求

**活动清单**：
- [ ] 召开需求调研会议（录音/纪要）
- [ ] 收集现有业务流程资料（Excel、文档、截图）
- [ ] 识别项目干系人和用户角色
- [ ] 初步估算项目规模和工期

**产出物**：
- `docs/需求调研笔记.md`
- 会议录音/录像（如有）

---

### 1.2 需求分析

**目标**：将收集的需求整理为结构化文档

**活动清单**：
- [ ] 整理业务流程图
- [ ] 识别功能模块和用户故事
- [ ] 识别非功能性需求（性能、安全、兼容性）
- [ ] 列出待确认问题清单
- [ ] 编写《业务流程与需求整理》文档

**产出物**：
- `docs/业务流程与需求整理.md`
- 业务流程图（Drawio/ProcessOn）
- 待确认问题清单

---

### 1.3 需求评审 ⭐️ 关键检查点

**目标**：与甲方确认需求理解无误

**触发条件**：
- 需求文档已完成
- 待确认问题清单已整理

**评审方式**：
- 会议评审（推荐）：与甲方召开正式评审会议
- 文档评审：发送文档给甲方书面确认

**评审检查清单**（参见 [references/requirement-review-checklist.md](references/requirement-review-checklist.md)）：
- [ ] 业务流程是否正确完整
- [ ] 功能模块是否遗漏
- [ ] 待确认问题是否都有答复
- [ ] 非功能性需求是否明确
- [ ] 项目边界是否清晰（哪些不做）

**产出物**：
- `docs/需求评审报告.md`
- 更新后的《业务流程与需求整理.md》（根据评审意见修订）

---

### 1.4 需求确认

**目标**：获得甲方正式确认，作为后续开发依据

**活动清单**：
- [ ] 将评审后的需求文档发送给甲方
- [ ] 获取甲方书面确认（签字/邮件/聊天记录）
- [ ] 冻结需求基线，建立变更控制流程

**产出物**：
- `docs/需求确认书.md` 或甲方确认邮件截图
- 需求基线版本（如：v1.0）

**⚠️ 重要**：未获得需求确认不得进入设计阶段

## 阶段2：设计阶段

### 2.1 原型/UI设计

**目标**：让客户可视化看到最终产品形态

**活动清单**：
- [ ] 设计信息架构和页面导航
- [ ] 制作低保真原型（线框图）
- [ ] 与甲方确认页面流程
- [ ] 制作高保真设计稿（视觉设计）
- [ ] 建立设计规范（颜色、字体、组件）

**工具推荐**：Figma、Axure RP、墨刀

**产出物**：
- 原型链接（Figma/Axure）
- `docs/UI设计规范.md`
- 关键页面设计截图

### 2.3 数据库设计

**目标**：设计完整的数据模型

**活动清单**：
- [ ] 识别核心实体和关系
- [ ] 设计表结构（字段、类型、约束）
- [ ] 设计索引策略
- [ ] 规划数据迁移脚本（如有历史数据）

**产出物**：
- `docs/系统设计/数据库设计.md`
- ER图
- 初始化SQL脚本

---

### 2.4 API接口设计

**目标**：定义前后端协作接口规范

**活动清单**：
- [ ] 梳理业务接口列表
- [ ] 定义请求/响应格式
- [ ] 定义错误码规范
- [ ] 编写接口文档

**产出物**：
- `docs/系统设计/API接口文档.md`
- 可选：Swagger/OpenAPI 文件

---

### 2.5 设计评审 ⭐️ 关键检查点

**目标**：内部评审设计方案，确保可实现

**评审检查清单**：
- [ ] 技术架构是否满足非功能性需求
- [ ] 数据库设计是否支持业务场景（特别是复杂查询）
- [ ] API设计是否RESTful、是否完整
- [ ] UI设计是否考虑移动端适配（如需要）
- [ ] 是否有性能瓶颈风险

**产出物**：
- `docs/设计评审报告.md`
- 更新后的设计文档

## 阶段3：开发阶段

### 3.1 开发环境搭建

**检查清单**：
- [ ] 代码仓库初始化（Git）
- [ ] 开发环境配置文档
- [ ] 本地开发环境可运行
- [ ] CI/CD流水线配置（如有）

**产出物**：
- `README.md`（项目说明）
- `docs/开发环境配置.md`

---

### 3.2 编码实现

**开发规范**：
- 遵循团队代码规范
- 编写单元测试（核心功能）
- 关键逻辑添加注释
- 定期提交代码（小步快跑）

**产出物**：
- 源代码
- 单元测试代码

---

### 3.3 代码审查

**审查重点**：
- [ ] 是否遵循代码规范
- [ ] 是否有明显性能问题
- [ ] 是否有安全隐患
- [ ] 是否有重复代码

**产出物**：
- 代码审查记录（GitHub/GitLab PR评论）

---

### 3.4 联调测试

**测试类型**：
- [ ] 接口联调（前后端）
- [ ] 功能测试（按需求逐项验证）
- [ ] 业务流程测试（端到端场景）
- [ ] 兼容性测试（不同浏览器/设备）

**产出物**：
- Bug清单和修复记录

---

### 3.5 测试报告 ⭐️ 关键检查点

**目标**：确认系统质量达到交付标准

**报告内容**：
- 测试范围（哪些功能已测/未测）
- Bug统计（严重/一般/轻微）
- 遗留问题和风险
- 是否通过测试的结论

**产出物**：
- `docs/测试报告.md`

**⚠️ 重要**：高优先级Bug未修复不得进入验收阶段

---

## 阶段4：验收阶段

### 4.1 内部验收

**检查清单**：
- [ ] 对照需求文档逐项验证
- [ ] 确认所有P0/P1功能已实现
- [ ] 确认已知问题已记录
- [ ] 准备演示环境

---

### 4.2 客户验收 ⭐️ 关键检查点

**目标**：客户确认系统满足需求

**验收方式**：
- 演示会议：向客户演示系统功能
- 试用验收：客户自行试用并反馈

**验收检查清单**（参见 [references/acceptance-checklist.md](references/acceptance-checklist.md)）：
- [ ] 核心业务流程是否通畅
- [ ] 数据是否正确显示和存储
- [ ] 客户提出的修改意见是否已处理
- [ ] 系统性能是否满足要求

**产出物**：
- `docs/验收报告.md`（客户签字确认）
- 遗留问题清单（如有）

**⚠️ 重要**：未通过客户验收不得上线

---

### 4.3 上线部署

**检查清单**：
- [ ] 生产环境已准备
- [ ] 部署文档已完成
- [ ] 数据迁移脚本已测试
- [ ] 回滚方案已准备
- [ ] 监控和告警已配置

**产出物**：
- `docs/部署文档.md`
- `docs/运维手册.md`
- 上线检查清单

## 变更控制

### 什么情况下需要变更控制？

- 需求文档已基线化（客户已确认）后
- 开发过程中客户提出新的需求或修改

### 变更流程

1. **提出变更**：记录变更内容、原因、影响范围
2. **评估影响**：评估工期、成本、技术影响
3. **决策**：与甲方协商是否接受变更
4. **执行**：更新文档、调整计划、实施变更
5. **确认**：更新后的需求重新确认

### 变更记录

- `docs/变更记录.md`

## 常见问题

**Q: 小项目是否也需要完整走这个流程？**  
A: 可以裁剪。小项目可以合并某些阶段，但关键检查点（需求评审、客户验收）不可省略。

**Q: 敏捷开发如何适配这个工作流？**  
A: 每个Sprint可以看作一个微型项目，在每个Sprint内遵循
